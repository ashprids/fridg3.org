# fridg3.org — Developer Reference (AI-Friendly)

This file explains how pages are rendered, where logic lives, how `/data` is structured, and where to implement specific changes safely.

---

## 1) Core Architecture

### Rendering model
- Most site routes are folder-based and served by `index.php` files.
- Each route generally follows:
  1. `session_start()`
  2. Set `$title`/`$description`
  3. Load root `template.html`
  4. Load route-local `content.html`
  5. Inject placeholders
  6. Echo final HTML
- Route `index.php` files use `find_template_file(...)` to walk upward until the requested file is found.

### Shared shell
- `template.html` is the global shell:
  - sidebar navigation
  - footer icons
  - `{user_greeting}` placeholder
  - `{content}` placeholder
  - includes `/style.css`, `/main.js`, Highlight.js, Font Awesome
- `main.js` is a large SPA-like enhancement layer (navigation interception, bookmark behavior, settings, toast bot UI, usage ASCII, etc.).
- `style.css` is global styling and contains reusable IDs/classes expected by many `content.html` files.

### Session and login state behavior
- Logged-in state is represented via `$_SESSION['user']`.
- Many pages replace footer account icon (`/account`) with logout icon (`/account/logout`) by string replacement in template HTML.
- Admin marker cookie `is_admin` is used by frontend WIP logic (not as sole backend authorization).

## 2) Global Entry Points

### `/` (root)
- `index.php` + `content.html`
- Injects dynamic homepage blocks:
  - `{latest_feed}` from `/data/feed/*.txt`
  - `{latest_journal}` from `/data/journal/*.txt`
  - `{latest_music}` from `/data/music/frdg3/*.json`
- Homepage currently renders 1 latest feed + 1 latest journal + up to 3 latest music cards.

### Root static/support files
- `template.html`: global shell and placeholders.
- `main.js`: frontend behavior and endpoint integration.
- `style.css`: all base styling.
- `sitemap.xml`: generated by API endpoint (`/api/sitemap`).

## 3) Page Groups (PHP + HTML)

## 3.1 Account pages

- `/account/index.php`
  - currently redirects to `/` if logged in, otherwise `/account/login`.
  - effectively not used as profile page right now.
- `/account/login/index.php` + `content.html`
  - CSRF token, session-hardening settings, login throttling via `/data/accounts/login_attempts.json`.
  - validates against `/data/accounts/accounts.json`.
  - sets `$_SESSION['user']` and `is_admin` cookie.
- `/account/logout/index.php`
  - destroys session + cookies and redirects to `/account/login?logged_out=1`.
- `/account/create/index.php` + `content.html`
  - admin-only account creation UI.
  - writes new accounts to `/data/accounts/accounts.json`.
- `/account/change-password/index.php` + `content.html`
- `/account/password/index.php` + `content.html`
  - both update logged-in user password hash in `/data/accounts/accounts.json`.
  - same behavior with slight template handling differences.

## 3.2 Feed pages

- `/feed/index.php` + `content.html`
  - list/search/paginate feed posts from `/data/feed/*.txt`.
  - writes `index.toml` as derived index.
  - hides create button unless admin or `allowedPages` includes `feed`.
- `/feed/create/index.php` + `content.html`
  - permission-gated creation.
  - supports image upload -> `/data/images`.
  - writes post text file to `/data/feed`.
  - can trigger Discord webhook from `/data/etc/webhooks.json` (`discord_feed`).
- `/feed/edit/index.php` + `content.html`
  - owner/admin editing and delete.
  - deleting a post also deletes referenced `/data/images/*` from `[img=/data/images/...]` BBCode.
- `/feed/posts/index.php` + `content.html`
  - single post view (`/feed/posts/{id}`).
  - derives OG image from first `[img=...]` tag.

## 3.3 Journal pages

- `/journal/index.php` + `content.html`
  - reads `/data/journal/*.txt`, search + pagination.
  - permission refresh from accounts.json into session.
  - create button visible to admin or `allowedPages` contains `journal`.
- `/journal/create/index.php` + `content.html`
  - permission-gated create + drafts.
  - BBCode-to-HTML conversion server-side.
  - uploads images to `/data/images`.
  - published posts to `/data/journal/{N}.txt`.
  - drafts to `/data/journal/drafts/*.txt` with owner line (`USER:<username>`).
- `/journal/posts/index.php` + `content.html`
  - single post renderer from `/data/journal/{id}.txt`.
  - treats body as trusted HTML.

## 3.4 Guestbook pages

- `/guestbook/index.php` + `content.html`
  - list + pagination from `/data/guestbook/*.txt`.
  - one-post-per-IP enforced via `/data/guestbook/ip_index.json`.
  - owner (IP-mapped) or admin can edit/delete.
- `/guestbook/create/index.php` + `content.html`
  - writes new entry (`timestamp + name + message`) to `/data/guestbook/*.txt`.
  - updates `ip_index.json`.
- `/guestbook/edit/index.php` + `content.html`
  - owner/admin edit flow with same file format.

## 3.5 Email + Newsletter pages

- `/email/index.php` + `content.html`
  - contact page; displays success notice via `?sent=1`.
- `/email/newsletter/index.php` + `content.html`
  - lists `/data/newsletter/*.html` as release archive.
  - create button shown only for admin.
- `/email/newsletter/create/index.php` + `content.html`
  - admin-only newsletter builder page shell.
- `/email/newsletter/create/preview/index.php` + `content.html`
  - admin-only preview shell.
- `/email/newsletter/preview/index.php` + `content.html`
  - general preview wrapper.
- `/email/newsletter/release/index.php` + `content.html`
  - reads one `/data/newsletter/{id}.html` and injects full HTML into page.

Mailing list status pages (mostly wrappers):
- `/email/mailinglist/subscribe`
- `/email/mailinglist/unsubscribe`
- `/email/mailinglist/invalid`
- `/email/mailinglist/error`

## 3.6 Music / Gallery / Saves / Settings / Others

- `/music/index.php` + `content.html`
  - builds `{frdg3_grid}` and `{cactile_grid}` from `/data/music/{artist}/*.json`.
- `/gallery/index.php` + `content.html`
  - image listing from `/data/images`, paginated.
  - admin delete controls call `/api/gallery/delete`.
- `/bookmarks/index.php` + `content.html`
  - renders saved feed/journal posts from account bookmarks.
- `/settings/index.php` + `content.html`
  - page shell; actual persistence via `/api/settings` and frontend JS.
- `/others/index.php` + `content.html`
  - wrapper page.
- `/others/off-topic-archive/index.php` + `content.html`
  - archive page wrapper; JS reads `/data/etc/off-topic-archive.json`.
- `/others/toast-discord-bot/index.php` + `content.html`
  - toast bot UI shell; frontend calls Discord-bot endpoints.

## 3.7 Formatting/demo + static errors

- `/formatting/index.php` + `content.html`
  - formatting showcase/settings helper page.
- `/formatting/example_page/index.php` + `content.html`
  - example skeleton.
- Static HTML error pages (not PHP-templated):
  - `/error/403/index.html`
  - `/error/404/index.html`
  - `/error/50x/index.html`
  - `/error/wip/index.html`

## 4) API Endpoints

- `/api/account/is-admin`
  - returns `{ isAdmin: boolean }`, refreshes `is_admin` cookie.

- `/api/bookmark`
  - POST only, requires logged-in user.
  - updates account `bookmarks` list in `/data/accounts/accounts.json`.
  - supports single toggle (`postId`) or replacement (`bookmarks[]`).

- `/api/feed-post`
  - GET by `?id=`.
  - reads `/data/feed/{id}.txt`, returns parsed JSON payload.

- `/api/gallery/delete`
  - POST admin-only.
  - deletes a filename from `/data/images` (validated extension/path).

- `/api/newsletter/publish`
  - POST (`id`, `html`) -> writes `/data/newsletter/{id}.html`.

- `/api/settings`
  - GET: fetch user settings from accounts.json.
  - POST: update glow intensity and colors (per-user in accounts.json).
  - POST (admin): update maintenance flag in `/data/etc/wip`.

- `/api/sitemap`
  - admin-only sitemap generator.
  - scans routes, feed posts, journal posts -> writes `/sitemap.xml`.

- `/api/system/usage`
  - CPU/memory/disk metrics (Linux + Windows logic paths).

- `/api/discord-bot-status`
  - reads `/data/etc/toast.json` and returns bot/stream status payload.

- `/api/discord-bot-control`
  - POST JSON `{url, name}` updates stream values in `/data/etc/toast.json`.
  - emits `.stream-update-signal` marker.

- `/api/discord-bot-control/status`
  - POST JSON `{status: "online"|"offline"}` updates bot status in `/data/etc/toast.json`.

- `/api/stream-proxy`
  - same-origin stream proxy for toast audio playback; host-restricted to configured stream host.

## 5) HTML Template Rules (What to edit where)

### `template.html`
Edit when changing:
- global sidebar/footer
- globally available controls
- global script/css includes
- `{title}`, `{description}`, `{user_greeting}`, `{content}` placeholders

### route `content.html`
Edit when changing:
- page-specific markup
- local placeholders consumed by page `index.php`

### route `index.php`
Edit when changing:
- server-side data loading/writing
- permissions/session checks
- placeholder injection
- redirects, query parsing, pagination, search

### `main.js`
Edit when changing:
- client-only behavior
- SPA nav interception
- settings UX, toast controls, bookmark toggles, archive rendering
- API fetch behavior

## 6) `/data` Directory Contract (Important)

`/data` is intentionally excluded from normal repo deployment workflows and should be managed separately.

## 6.1 `/data/accounts/`

### `accounts.json`
Expected structure:
```json
{
  "accounts": [
    {
      "username": "string",
      "name": "string",
      "password": "bcrypt-hash or empty",
      "isAdmin": true,
      "allowedPages": ["feed", "journal"],
      "bookmarks": ["2026-01-01_12-00-00", "journal:12"],
      "glowIntensity": "none|low|medium|high",
      "colors": {
        "bg": "#RRGGBB",
        "fg": "#RRGGBB",
        "border": "#RRGGBB",
        "subtle": "#RRGGBB",
        "links": "#RRGGBB"
      }
    }
  ]
}
```

### `login_attempts.json`
- object map of client-IP to unix timestamp arrays.
- used for login throttling.

## 6.2 `/data/feed/`
- Post file format (`*.txt`):
  1. `@username`
  2. datetime (`YYYY-MM-DD HH:MM:SS`)
  3. body text (optional BBCode)
- `index.toml` is generated by `/feed/index.php` (derived artifact, can be regenerated).

## 6.3 `/data/journal/`
- Published post format (`N.txt`):
  1. date (`YYYY-MM-DD`)
  2. title
  3. subtitle/description
  4+. trusted HTML body

- Draft format (`/data/journal/drafts/*.txt`):
  1. `USER:<username>`
  2. title
  3. description
  4+. BBCode body

## 6.4 `/data/guestbook/`
- Entry file (`*.txt`):
  1. timestamp (`YYYY/MM/DD HH:MM:SS`)
  2. display name
  3+. message text
- `ip_index.json`: maps client IP => one guestbook filename.

## 6.5 `/data/images/`
- Uploaded images used by feed/journal/gallery/newsletters.
- Expected web path: `/data/images/<filename>`.

## 6.6 `/data/music/`
- Artist folders currently: `frdg3/`, `cactile/`.
- Album JSON format:
```json
{
  "album_name": "string",
  "album_caption": "string",
  "album_type": "Album|EP|Single|Remix|...",
  "album_art": "/data/images/example.jpg",
  "album_art_directory": "/data/images/example.jpg",
  "order": 6,
  "songs": [
    { "name": "Track name", "directory": "/data/audio/file.wav" }
  ]
}
```
- `album_art_directory` is preferred by code; `album_art` is still supported.

## 6.7 `/data/audio/`
- Audio assets referenced by music album `songs[].directory`.
- Mini player and toast listen-along logic may stream from these paths.

## 6.8 `/data/newsletter/`
- Each release saved as `{id}.html` (typically date-like `YYYY-MM-DD.html`).
- Used by `/email/newsletter` listing and `/email/newsletter/release/{id}` rendering.

## 6.9 `/data/etc/`

### `wip`
- plain text flag (`true`/`false` or equivalent truthy/falsey forms).
- consumed by `main.js` maintenance guard.

### `webhooks.json`
Expected keys used by code:
```json
{
  "discord_feed": "https://discord.com/api/webhooks/..."
}
```

### `toast.json`
Expected keys:
```json
{
  "bot": { "token": "...", "client_id": "...", "status": "online|offline" },
  "stream": { "url": "http(s)://...", "name": "..." },
  "channel": { "id": "...", "name": "..." },
  "features": { "auto_play": true, "loop": true }
}
```

### `toast-updates.json`
- array of status history entries:
```json
[{ "time": "YYYY-MM-DD HH:MM:SS", "status": "..." }]
```

### `off-topic-archive.json`
- large Discord export object with `guild`, `channel`, `messages[]`.
- frontend archive viewer expects message-level fields like `timestamp`, `content`, `author`, `attachments`, etc.

## 6.10 `/data/downloads/`
- downloadable artifacts surfaced by pages/links (e.g. binaries, preset files).

## 7) Development Playbook (How to make safe changes)

1. Identify route type:
   - wrapper route (simple template/content injection) vs data-backed route.
2. If visual-only change:
   - edit route `content.html` (and `style.css` only if global style needs adjustment).
3. If data/permission/change in behavior:
   - edit route `index.php` and possibly API endpoint + `main.js`.
4. If cross-site UI change:
   - edit `template.html` and verify footer/sidebar replacements still match literal strings used in PHP.
5. If new persisted field is needed:
   - update write path + read path + default handling in `/data` schema.
6. For admin-only behavior:
   - enforce in PHP endpoint/page, not only frontend.

---

## 8) Known Couplings / Gotchas

- String-based template replacement is fragile:
  - many pages rely on exact HTML of the account footer button to swap login/logout.
- Bookmark storage currently uses `/data/accounts/accounts.json`.
  - some older logic references `/data/users` patterns; keep bookmark source consistent with accounts.json unless intentionally migrating.
- Journal posts store HTML body directly; feed posts store raw text/BBCode.
- Newsletter publish endpoint is under `/api/newsletter/publish` (not under `/email/newsletter/publish`).
- `main.js` is large and route-sensitive; always re-test impacted route transitions when changing shared JS.

## 9) File Coverage Checklist

The following are the primary files that carry behavior and should be reviewed before major edits:
- Root: `index.php`, `template.html`, `content.html`, `main.js`, `style.css`
- Account: all `account/**/index.php` + corresponding `content.html`
- Feed: `feed/index.php`, `feed/create/index.php`, `feed/edit/index.php`, `feed/posts/index.php` + all feed `content.html`
- Journal: `journal/index.php`, `journal/create/index.php`, `journal/posts/index.php` + all journal `content.html`
- Guestbook: `guestbook/index.php`, `guestbook/create/index.php`, `guestbook/edit/index.php` + all guestbook `content.html`
- Email/Newsletter: `email/**/index.php` + all `email/**/*.html` templates
- Music/Gallery/Bookmarks/Settings/Others: route `index.php` + route `content.html`
- API: all `api/**/index.php`
- Static error pages: `error/403/index.html`, `error/404/index.html`, `error/50x/index.html`, `error/wip/index.html`

If an AI is asked to change a page, it should first classify whether that page is:
- static template wrapper,
- server-rendered data page,
- API-backed JS page,
then edit the smallest correct surface.
